Backend Folder Reference
Purpose, responsibilities, and key files for each backend directory
⚙️ lib/ mvp-backend/lib/
The infrastructure layer. Contains all configuration, authentication logic, role definitions, and shared utilities. Every other layer depends on lib/ — it depends on nothing else in the codebase.
Key files
supabase_client.py — Initializes the Supabase Python client using env vars. Exports a singleton supabase instance used by all services.
auth.py — Defines get_current_user as a FastAPI Dependency that decodes JWT from the Authorization header via Supabase. All protected routes inject this.
roles.py — Enums for PassionRole, SubchannelRole, and EventOrganizerPermission. Centralizes role hierarchy logic.
permissions.py — FastAPI dependency factories like require_passion_role("admin") that check membership + role before allowing access.
exceptions.py — Custom exceptions: NotFoundError, ForbiddenError, DuplicateError. Mapped to HTTP status codes via exception handlers in main.py.
📐 schemas/ mvp-backend/schemas/
Pydantic models that validate every byte entering and leaving the API. Each domain entity gets a schema file with Create, Update, and Response variants. Schemas never import from routers or services — they are pure data contracts.
Pattern per file
*_schema.py — Contains EntityCreate (input), EntityUpdate (partial input), EntityResponse (output with computed fields), and any nested sub-models.
common_schema.py — Shared models: PaginationParams(offset, limit), PaginatedResponse[T], StatusResponse(success, message).
🛤️ routers/ mvp-backend/routers/
Thin HTTP layer. Each router file defines endpoints for one domain, injects auth dependencies, validates input via schemas, and delegates all logic to the corresponding service. Routers never touch the database directly.
Anatomy of a router
*_router.py — Creates an APIRouter(prefix="/entity", tags=["Entity"]). Each endpoint is a decorated async function that calls entity_service.method() and returns a schema response.
🧠 services/ mvp-backend/services/
The brain. Contains all business logic, database queries (via Supabase client), data transformations, permission enforcement, and side effects (e.g., creating a notification when someone likes a post). Services import from lib/ and schemas/ only.
Responsibilities
*_service.py — Exports async functions: create_*(), get_*(), update_*(), delete_*(). Each function receives typed params, queries Supabase, applies business rules, and returns data or raises exceptions.
🗄️ migrations/ mvp-backend/migrations/
Versioned, sequential SQL files that define and evolve the database schema. Prefixed with a 3-digit number for ordering. Run against Supabase via the SQL editor or a migration runner script. Never modify an existing migration — create a new one.
Naming pattern
NNN_description.sql — e.g., 001_create_users.sql, 010_add_passion_tags.sql. Each file is idempotent (uses IF NOT EXISTS) and includes both the table creation and its indexes/constraints.
🚀 main.py mvp-backend/main.py
The application entry point. Creates the FastAPI instance, registers all routers with app.include_router(), configures CORS middleware, and registers the custom exception handlers from lib/exceptions.py. This is the only file that imports from routers/.

Frontend Folder Reference
Purpose, responsibilities, and key files for each frontend directory
🔧 config/ src/config/
Environment configuration. A single file that reads environment variables and exports typed constants. Nothing else in the app reads process.env directly — everything goes through config/env.ts.
Files
env.ts — Exports API_BASE_URL, SUPABASE_URL, SUPABASE_ANON_KEY. Uses fallbacks for dev vs production.
🔌 lib/ src/lib/
Low-level infrastructure clients. These are the plumbing — consumed by api/, context/, and hooks/, but never by screens or components directly.
Files
apiClient.ts — Axios instance with base URL from config, request interceptor that attaches the JWT from secure storage, and response interceptor for 401 → auto-refresh logic.
supabase.ts — Initializes createClient() with custom storage adapter that delegates to secureStorage.ts instead of AsyncStorage for token persistence.
secureStorage.ts — Wraps react-native-keychain with a SERVICE = 'supabase-auth' namespace. Exports getToken(), setToken(), clearToken().
🔐 context/ src/context/
React Context providers that hold global state. Currently only AuthContext, but can grow to include ThemeContext or NotificationContext as needed.
Files
AuthContext.tsx — Provides user, session, isLoading, login(), logout(), signup(). Listens to Supabase onAuthStateChange and persists tokens via secureStorage.
📝 types/ src/types/
TypeScript interfaces that mirror the backend Pydantic schemas. These are the single source of truth for data shapes on the frontend. Every API function and component prop should reference these types.
Pattern
entity.ts — Exports Entity (full response type), EntityCreate (input), EntityUpdate (partial). Field names match the backend JSON keys exactly (camelCase from FastAPI's alias_generator).
📡 api/ src/api/
Typed async functions that call the backend via apiClient. One file per domain. These are the only files that import apiClient — hooks and screens never make HTTP calls directly.
Pattern
entityApi.ts — Exports named functions like getEntity(id: string): Promise<Entity>. Each function calls apiClient.get/post/patch/delete and returns typed data.
🪝 hooks/ src/hooks/
Custom React hooks that encapsulate state management, API calls, caching, and side effects. Screens consume hooks — they never call API functions directly. This gives you a clean layer to add caching, optimistic updates, or offline support later.
Examples
useAuth.ts — Convenience wrapper around useContext(AuthContext).
useFeed.ts — Manages feed state, pagination, pull-to-refresh, phriends-only filter toggle. Calls postApi.getFeed().
🎨 theme/ src/theme/
The design system. Defines colors (light + dark), typography scales, spacing tokens, and a unified useTheme() hook. Components import from theme — they never hardcode colors or font sizes.
🧩 components/ src/components/
Reusable UI building blocks organized into subfolders by purpose: common/ for primitives (Button, Input, Avatar), cards/ for data display cards, forms/ for creation forms, layout/ for structural elements. Components are presentation-focused and receive data via props.
📱 screens/ src/screens/
Full-page views, one per navigation route. Screens are organized into feature subfolders (auth, feed, passions, events, social, profile, notifications). Each screen composes components + hooks to build the complete UI. Screens are the only files registered in the navigator.
🧭 navigation/ src/navigation/
React Navigation configuration. AppNavigator.tsx defines the stack/tab structure and conditionally renders auth vs main flows based on useAuth(). types.ts exports RootStackParamList so every navigation.navigate() call is type-safe.
📦 assets/ src/assets/
Static resources: images (logos, icons, placeholders), Lottie animations (loading, success, empty states), and custom fonts. Subfoldered by type. Referenced via require() or asset modules.

File Creation Decision Rules

📐 Adding a New Domain Entity

When you add a new table to the database (e.g., tags), you must create files across all layers in this exact order:

1. migrations/010_create_tags.sql 2. schemas/tag_schema.py 3. services/tag_service.py 4. routers/tag_router.py 5. main.py → add include_router(tag_router) --- 6. src/types/tag.ts 7. src/api/tagApi.ts 8. src/hooks/useTags.ts 9. src/screens/tags/TagScreen.tsx 10. src/navigation/types.ts → add route params

🧩 Adding a Reusable Component

Before creating a new component, check if an existing one can be extended with props. New components go in the appropriate subfolder based on their purpose.

Primitive (button, input, toggle)? → components/common/ComponentName.tsx Data display card? → components/cards/EntityCard.tsx Form for entity creation/edit? → components/forms/CreateEntityForm.tsx Layout/structural? → components/layout/ComponentName.tsx

🛤️ Adding a New API Endpoint

A new endpoint always requires changes in three backend files. If it's a new HTTP method on an existing entity, add to the existing files. If it's a new sub-resource, consider whether it belongs in the parent's files or needs its own.

Backend: schemas/entity_schema.py → add input/output model services/entity_service.py → add business logic routers/entity_router.py → add route decorator Frontend: api/entityApi.ts → add typed function hooks/useEntity.ts → expose to screens

🗄️ Adding a Migration

Never edit existing migrations. Always create a new file with the next sequential number. Each migration is additive.

Next available number: check last file number + 1 Name format: NNN_verb_noun.sql Verbs: create, alter, add, drop, create_index Examples: 010_add_passion_tags.sql 011_create_index_posts_feed.sql 012_alter_events_add_timezone.sql

📱 Adding a New Screen

Every screen belongs in a feature subfolder under screens/. After creating the file, register it in the navigator and add its params to the type definitions.

1. screens/feature/FeatureScreen.tsx 2. navigation/types.ts → add to RootStackParamList 3. navigation/AppNavigator.tsx → add <Stack.Screen>

⛔ What NOT to Do

These anti-patterns break the architecture's separation of concerns and must be avoided.

✗ Screens calling apiClient directly → use hooks, which use api/ functions ✗ Routers containing business logic → delegate everything to services ✗ Components importing from api/ → pass data via props from screens ✗ Hardcoded colors or font sizes → import from theme/ ✗ Editing existing migration files → create a new sequential migration ✗ Services importing from routers → dependencies flow downward only
